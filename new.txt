

### 1. Objective of the Game

The game simulates a **two-party negotiation scenario**, based on the original experiment **“Fahrrad-Verhandlungsspiel”**.

Two people, **A and B**, each own one part of a bicycle.
Only by **cooperating** can they sell the complete bicycle worth **€1,000**.

They must negotiate to decide **how to divide the €1,000**.

The objective is to **reach an agreement – or fail if no agreement is reached**.

---

### 2. Introduction Screen

The start screen displays the following introduction:

**Welcome to the Bicycle Negotiation Game!**

Two people (A and B) each own one component of a bicycle:

Person A: owns the bicycle wheels (€200)
Person B: owns the bicycle frame (€600)

Only together can they assemble a complete bicycle and sell it for **€1,000**.

They must decide **how the €1,000 should be divided between them** if a negotiation takes place.

For this negotiation, both parties also receive **an alternative selling option**.
If the negotiation fails, this alternative option is automatically activated.

After this text appears a button:

👉 **“Start the Game”**

---

### 3. Role Assignment and Pair Identification

#### Automatic Role Assignment

When entering the game, the system automatically assigns roles:

One player becomes **Person A**
One player becomes **Person B**

Each player receives a **unique player ID** (for example: `Player_0123`).

The system randomly matches two players into **a negotiation pair**.

---

#### Pair Identification Code

Each pair receives a **Pair-ID** (for example: `Pair_01`, `Pair_02`, etc.).

This ID is used to identify **which two players negotiated with each other**.

---

### 4. Game Groups (Groups 1–4)

There are **four separate groups**, each defining a different **alternative selling option for Person B**.

Players **do not know the real economic meaning (BATNA)**.
They only see the sentence:

**“Person B has the alternative selling option … €”**

| Group | Person A | Person B |
| ----- | -------- | -------- |
| 1     | 0 €      | 0 €      |
| 2     | 0 €      | 300 €    |
| 3     | 0 €      | 500 €    |
| 4     | 0 €      | 600 €    |

Person A always has a **starting value of 0 €**.

Person B only knows **the minimum amount they expect**, but not the term **BATNA**.

Group assignment happens **before the game begins**.

---

### 5. Negotiation Process

#### Type of Game

The game simulates **a turn-based bilateral negotiation**.

Person A and Person B **take turns making offers and responding**.

Maximum number of rounds: **10 rounds**

---

### Structure of a Negotiation Round

| Step | Action                           | Description                                   |
| ---- | -------------------------------- | --------------------------------------------- |
| 1    | Player whose turn it is (A or B) | Enters an offer, e.g. “A: 400 €, B: 600 €”    |
| 2    | Other player                     | Chooses one of four responses                 |
| 3    | Round result                     | Determines whether the game continues or ends |

Response options:

☐ Too low
☐ Accept
☐ Better offer
☐ Not accept

If **Accept** or **Not accept** → the game ends.
If **Too low** or **Better offer** → the turn switches.

---

### End Conditions

One player chooses **Accept** → negotiation successful.

One player chooses **Not accept** → negotiation fails.

If **10 rounds pass without agreement** → automatic failure.

---

### 6. Response Options (Actions)

The four responses from the original experiment:

**Too low** – That is too low for me, counteroffer.

**Accept** – The offer is accepted, end of game.

**Better offer** – I have a better offer outside the negotiation.

**Not accept** – Negotiation is terminated, end of game.

---

### 7. Final Result Screen

If successful:

Negotiation successful!

Final distribution:

Person A: ___ €
Person B: ___ €

If failed:

Negotiation failed.
No agreement reached.

Person A receives **0 €**.

Person B receives **their minimum demand**
(0 / 300 / 500 / 600 € depending on the group).

---

Each action is recorded in the corresponding **round**.

When the game ends, the system automatically **exports all data to an Excel file (.xlsx)**.

Each row in the Excel file represents **one negotiation round**.
